You are an expert in reversible computing and the reversible programming language Janus.

Write a Janus procedure with the following signature:

    procedure rabinkarp1(int T[], int n, int P[], int m, int d, int q, stack r)

The procedure should: Implement the reversible 1D Rabin-Karp string matching algorithm. Given a text T (n symbols) and a pattern P (m symbols), find every start position s with 0 <= s <= n-m where the length-m window of T starting at s equals P, and push each such s onto the stack r in increasing order of s (so r ends holding all match positions). Use a modular polynomial rolling hash with radix d and prime modulus q: the hash of a window is its base-d value mod q; slide the window updating the hash incrementally, and on a hash match confirm with a direct (reversible) character comparison before pushing s. T, n, P, m, d and q must be unchanged; r starts empty. Compute and then uncompute every intermediate hash and temporary so that no garbage remains and the whole procedure is reversible. Signature: procedure rabinkarp1(int T[], int n, int P[], int m, int d, int q, stack r). Assume q is prime and gcd(d,q)=1. Example: T=[1,2,1,2,1,2,1,2,1,2], n=10, P=[1,2,1,2], m=4, d=26, q=3 -> r = <6, 4, 2, 0] (matches at 0,2,4,6).  You MUST implement this with the modular polynomial rolling hash described above: multiply by the radix d, reduce modulo the prime q, and update the hash incrementally as the window slides, confirming a match by direct comparison only when the hashes agree, and uncompute the hashes. A naive character-by-character comparison that does not compute the hash is NOT accepted, even if it produces the correct match positions. 

Example: when called with T=[1, 2, 1, 2, 1, 2, 1, 2, 1, 2], n=10, P=[1, 2, 1, 2], m=4, d=26, q=3, the expected result is P=[1, 2, 1, 2], T=[1, 2, 1, 2, 1, 2, 1, 2, 1, 2], d=26, m=4, n=10, q=3, r=[6, 4, 2, 0].

Output rules:

The BNF of the Janus language is as follows:
```
<Program> ::= (<ProcedureDef>)* <MainProcedureDef>
<ProcedureDef> ::= "procedure" <IdentNonMain> "(" [ <Param> ("," <Param>)* ] ")" (<Stmt>)+
<MainProcedureDef> ::= "procedure" "main" "(" ")" (<MainVdecl>)* (<Stmt>)+

<Param> ::= "int" <Ident> [ "[" "]" ]

<MainVdecl> ::= "int" <Ident> [ "=" <Expr> ]  | "int" <Ident> "[" "]" [ "=" <ArrayLit> ]
<ArrayLit> ::= "{" <Expr> ("," <Expr>)* "}"

<LocalDecl> ::= "int" <Ident> "=" <Expr>
<LocalDeclList> ::= <LocalDecl> ("," <LocalDecl>)*

<Stmt>
  ::= <Lval> <ModOp> <Expr>
   | "if" <Expr> "then" (<Stmt>)+ "else" (<Stmt>)+ "fi" <Expr>
   | "from" <Expr> "do"   (<Stmt>)+ "loop" (<Stmt>)+ "until" <Expr>
   | "local" <LocalDeclList> (<Stmt>)+ "delocal" <LocalDeclList>
   | "call" <Ident> "(" [ <ArgList> ] ")"
   | "uncall" <Ident> "(" [ <ArgList> ] ")"
   | <Lval> "<=>" <Lval>
   | "skip"

<ModOp> ::= "+=" | "-=" | "^=" | "*=" | "/="

<ArgList> ::= <Expr> ("," <Expr>)*

<Lval> ::= <Ident> [ "[" <Expr> "]" ]

<Expr> ::= <Expr> <BinOp> <Expr> | ("!" | "~") <Expr> | <Term>
<Term> ::= "(" <Expr> ")" | <Integer> | <Lval>
<BinOp> ::= "*" | "/" | "%" | "+" | "-" | "<=" | "<" | ">=" | ">" | "=" | "!=" | "^" | "&&" | "||"

<Ident> ::= <letter> (<alphaNum> | "_" | "'")*
<IdentNonMain> ::= <Ident>
<Integer> ::= <decimal-integer>
```

Static Reversibility Constraints (well-formedness rules)

```
C1. Reversible assignment (non-self-referential update)
    For any statement:
        <Lval> <ModOp> <Expr>
    let Base(<Lval>) be the identifier at the root of the lval
    (i.e., Base(x)=x, Base(a[i])=a).
    Then Base(<Lval>) MUST NOT occur in <Expr> as an identifier
    (neither as a variable nor as an array base).

          Examples:
            OK:   i += 1
            OK:   a[i] += 1
            NG:   i += i
            NG:   a[i] += a[j]

C2. If/fi consistency (entry/exit condition discipline)
    For:
        if E_in then S1 else S2 fi E_out
    forward execution must satisfy:
      - If E_in is true: execute S1 and after S1, E_out must be true.
      - If E_in is false: execute S2 and after S2, E_out must be false.
    (In particular, E_in and E_out must be chosen so that the branch taken
     forward can be reconstructed backward.)

C3. From/until loop consistency + execution order (strengthened)
    For:
        from E_in do S1 loop S2 until E_out

    Execution order (forward):
      1) Evaluate E_in. It MUST hold at loop entry.
      2) Execute S1 once (if absent, treat as skip).
      3) Evaluate E_out.
         - If E_out is true, terminate the loop immediately.
         - If E_out is false, execute S2 once (if absent, treat as skip),
           then evaluate E_in and it MUST be false, and go back to step (2).

C4. local/delocal discipline (static well-formedness)

    For any block:
        local D1, ..., Dk   S   delocal D1', ..., Dk'

    the following must hold:

      (a) Arity agreement:
          The number of declarations must be the same:
              k = k'.

      (b) Identifier agreement:
          The declared identifiers must match exactly,
          in the same order and with the same names.
          (Only the initializer expressions may differ.)

      (c) Scope discipline:
          The identifiers introduced by the local declarations
          are in scope only within S.
          They must not be accessed outside the block.

    Intuition:
      The local/delocal construct introduces temporary variables
      whose lifetimes are statically delimited,
      while the delocal declarations specify the values
      required to re-enter the block during backward execution.

C5. Scope / declared-identifier discipline
    A program may use only identifiers that are in scope.
      - In a procedure body: identifiers must be procedure parameters or locals
        introduced by an enclosing local/delocal block.
      - In main: identifiers must be declared by earlier <MainVdecl> or locals.
      - No implicit globals.
      - No shadowing: a local declaration MUST NOT reuse an identifier already in scope.

C6. delocal expected-value check (runtime)
    For:
        local int x = E_in   S   delocal int x = E_out
    runtime check at delocal:
      - Immediately before delocal, the implementation checks:
            x == Eval(E_out)
        If it fails, abort.
    Note: E_out need not equal E_in.

      Examples:
            NG:   E_in is x
            NG:   E_out is x

C7. Semantics of the reversible update operators:
    - `x *= e` multiplies x by e and is undone by `x /= e`; it requires e != 0.
    - `x /= e` requires e != 0 and that x is exactly divisible by e (no remainder).
```
